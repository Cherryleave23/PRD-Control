# spec-modeling —— 动态记录 + 软提醒防偏离

主动构建并维护项目的需求/边界模型。这是 spec-clarify 编排器在阶段 ②③④ 调用的核心纪律，也是 spec-grilling 访谈时联动的写入目标。

## 双重职责

- **访谈期（阶段 ①②）**：主动构建并锐化项目的需求/边界模型——挑战术语、发明边缘场景、把敲定的内容立即写入文档。仅仅"读" `CONTEXT.md` 不算本 skill 的职责——本 skill 用于"改变"模型，不是"消费"模型。
- **实现期（阶段 ③④）**：对照 `SPEC.md` 检查实现偏离，按偏离类型分级处理（违反不变式硬阻断自动返工，其他三类软提醒）；发现 SPEC 本身有误时记录规范偏离并触发状态回退；收尾时同步文档状态。

## 文件结构

```
/
├── SPEC.md                    # 需求 + 边界 + 不变式 + 验收标准
├── CONTEXT.md                 # 术语表（glossary）
├── docs/
│   └── adr/
│       ├── 0001-{slug}.md
│       └── 0002-{slug}.md
└── src/                       # 业务代码
```

**多上下文项目**用 `CONTEXT-MAP.md` 映射（格式见 `CONTEXT-FORMAT.md`）。

**懒创建**：只在有内容写时才创建文件。若没有 `SPEC.md`，在第一条需求敲定时创建；若没有 `docs/adr/`，在第一个 ADR 需要时创建。

## 访谈期行为

### 对照 glossary 挑战术语冲突

用户用词与 `CONTEXT.md` 冲突时立即指出。例："你的 glossary 定义'取消'为 X，但你似乎在说 Y——到底是哪个？"

### 锐化模糊语言

用户用模糊或重载词时，提议精确的规范术语。例："你说的'账户'——指客户还是用户？这是两回事。"

### 讨论具体场景

领域关系被讨论时，构造边缘场景压力测试，迫使用户精确划定概念边界。

### 与代码交叉验证

用户陈述"某事如何工作"时，检查代码是否一致。矛盾则浮现："你的代码取消整个订单，但你刚说支持部分取消——哪个对？"

### 内联更新

术语/需求/边界/不变式敲定即写，不批量。`CONTEXT.md` 只放术语，绝不放实现细节——它不是 spec、不是草稿本。

### 可证伪性自检（写入不变式前）

写入不变式 I-x 前，做轻量表述质量检查：**能写出返回 true/false 的检查函数吗？**

- 能 → 用 `> 自检：\`{伪代码}\`` 格式附一段自检伪代码在不变式条目后，然后写入。伪代码不强制存放路径、不强制运行——目的是让检查语义从"脑内自问"外化为肉眼可见。
- 不能（如"系统应当合理""代码应当优雅"）→ 提示用户"表述不可证伪，建议重写为可检查的形式"，但仍写入（不阻断，保留用户控制权）。此时不附伪代码。

**只做可证伪性自检，不重新做分类决策**——分类已由 `spec-grilling` 与用户在访谈中确定（用精炼三问，详见 `SPEC-FORMAT.md` 的"写入前自检"）。本自检只兜底表述质量，不改分类。

## SPEC.md 三段式（核心扩展点 1）

`SPEC.md` 是本生态相对 domain-modeling 的核心扩展，三段式结构（详见 `SPEC-FORMAT.md`）：

1. **需求（Requirements）**：用 EARS 句式表达，每条独立可验，编号便于引用。
   - 无处不在型：系统应当...
   - 状态驱动型：当 [状态]，系统应当...
   - 事件驱动型：当 [事件] 发生，系统应当...
   - 不期望型：如果 [错误条件]，系统应当...
2. **边界（Boundaries）**：显式列出范围内与范围外。
   - 范围内：...
   - 范围外（不做）：... ← 显式的 no 和显式的 yes 同样有价值
3. **不变式（Invariants）**：系统在任何情况下都要满足的条件，无论外部输入如何。例："订单总额 = 各行小计之和 - 折扣 - 优惠，始终非负。"
4. **验收标准（Acceptance）**：给定/当/那么 句式，可映射为测试。

`SPEC.md` 顶部维护状态字段：`draft | aligned | implementing | verified | superseded`。

## 偏离防偏离机制（核心扩展点 2）

### 实现前预检

开始写实现代码前，列出本次将触及的 SPEC 条目（需求/边界/不变式），明确"本次实现承诺满足哪些条款"。这让偏离有对照基准。

### 偏离识别与分级

每完成一个可验证单元，对照 SPEC 条目检查四类偏离：

- **越界**：是否实现了 SPEC 未要求的东西？
- **缺斤少两**：是否漏实现 SPEC 要求的东西？
- **违反不变式**：是否破坏了系统在任何情况下都要满足的约束？
- **触碰"范围外"清单**：是否实现了显式标注"不做"的功能？

识别后按类型分级处理（决策 #2）：

| 偏离类型 | 处理方式 |
|---|---|
| **违反不变式**（=与设计边界完全相反） | **硬阻断**——agent 自动返工，**不提示操作者** |
| 越界 / 缺斤少两 / 触碰范围外 | **软提醒**——提示用户 + 给建议，继续执行 |

**硬阻断例外**（详见 `docs/adr/0001-硬阻断例外.md`）：不变式是系统在任何情况下都要满足的约束，违反它意味着实现与设计边界完全相反，系统进入未定义状态。此场景下处置权交给 agent，确保系统约束不被轻易破坏。这是"不阻断"原则的唯一例外，边界严格限定为"违反不变式"。

### 软提醒的根因×后果诊断

对软提醒类偏离（越界/缺斤少两/触碰范围外），执行二维诊断：

**Step 1 — 识别偏离类型**（事实，自查代码）
越界 / 缺斤少两 / 触碰范围外

**Step 2 — 诊断根因**（事实，自查代码）

**Step 3 — 评估后果**（事实，运行/测试）
偏离后该功能是否仍能保持正常运作？

**Step 4 — 查矩阵给建议**（决策，由用户拍板）

根因×后果矩阵与偏离分级表在 `DEVIATIONS-FORMAT.md`（单一真相源，I10）。Step 1-3 是**事实**，自己查代码/运行测试判断，不问用户（遵循 grilling 的"事实查环境"）。Step 4 是**决策**，查矩阵给建议，用户拍板（遵循"决策问用户"）。

**硬阻断例外**：违反不变式（任意根因/任意后果）直接自动返工，不走矩阵，不提示操作者（详见 `docs/adr/0001-硬阻断例外.md`）。

**强提醒仍不阻断**——只是风险可见性更高 + 建议更明确，最终决定权仍在用户。硬阻断只针对违反不变式。

### 规范偏离（SPEC 本身有误）

实现期发现 SPEC 条目本身有误（非实现错误，是规范错误）时：

1. **状态回退**：`SPEC.md` frontmatter 的 `status` 从 `implementing` 改为 `draft`。
2. **记录规范偏离**：在 `docs/deviations.md` 记一条，类型标"规范偏离"，根因说明哪条条款错了、为什么。
3. **保留已写代码**：不自动删除——在偏离记录中标注"待重评估"。
4. **回阶段 ①**：`spec-grilling` 只针对冲突条款重新访谈，不推翻整个 SPEC。重访后决定代码是保留、修改还是删除（决策权在用户）。

### 偏离记录归档（单一真相源）

所有偏离记录（实现偏离 + 规范偏离）统一存放到 `docs/deviations.md`——**这是偏离记录的唯一存放点**（不变式 I3）。SPEC.md、CONTEXT.md、ADR 均不得含偏离记录段落。

每条记录含六字段（详见 `DEVIATIONS-FORMAT.md`）：

- **类型**：违反不变式 | 越界 | 缺斤少两 | 触碰范围外 | 规范偏离
- **根因**：实现意外 | 返工缺陷 | 模型自添加设计 | 其他
- **后果**：功能正常 | 功能异常
- **处理决定**：自动返工 | 软提醒+用户决定 | 状态回退
- **日期**：YYYY-MM-DD
- **待评估代码位置**：文件路径，可选，仅规范偏离时填写

## 节制地提供 ADR

仅当三条件全满足才提议写 ADR：

1. **难以撤销**——改变主意成本显著
2. **无背景则令人惊讶**——未来读者会疑惑"为什么这么做"
3. **真实权衡的结果**——存在真实备选方案，因具体理由选了其一

任一缺失则跳过。格式见 `ADR-FORMAT.md`。

## 收尾同步（阶段 ④）

实现完成时：

- 更新 `SPEC.md` 的 YAML frontmatter `status` 字段（`implementing` → `verified` 或 `superseded`）——**只改 frontmatter，不改正文**，且只解析 frontmatter 判断状态。
- 补充实现期浮现的新决策到 `docs/adr/`。
- 修订 `CONTEXT.md` 术语（新增的实现期术语补入，冲突的修正）。
- 审查 `docs/deviations.md` 偏离清单，决定哪些需更新 SPEC、哪些归档。

## 引用格式模板

- 写 `SPEC.md` 用 `SPEC-FORMAT.md`（frontmatter 字段集固定为 `status`/`last-updated`/`topic`，不得增减）
- 写 `CONTEXT.md` 用 `CONTEXT-FORMAT.md`
- 写 `docs/adr/` 用 `ADR-FORMAT.md`
- 写 `docs/deviations.md` 用 `DEVIATIONS-FORMAT.md`（偏离记录唯一存放点）
