# ADR 格式

ADR（Architecture Decision Record，架构决策记录）存放在 `docs/adr/`，由 `spec-modeling` 在访谈期或实现期节制地创建。

## 存放与编号

- 目录：`docs/adr/`
- 文件名：`0001-{slug}.md`、`0002-{slug}.md`，顺序编号
- **懒创建**：只在第一个 ADR 需要时才创建 `docs/adr/` 目录
- **编号**：扫描 `docs/adr/` 取最大编号 +1

## 何时提议 ADR（三条件全满足）

仅当三条件全满足才提议写 ADR：

1. **难以撤销**——改变主意成本显著
2. **无背景则令人惊讶**——未来读者看代码会疑惑"为什么这么做"
3. **真实权衡的结果**——存在真实备选方案，因具体理由选了其一

任一缺失则跳过。不满足三条件的决策不写 ADR——避免 ADR 膨胀。

## 模板（极简版）

```markdown
# {决策的简短标题}

{1-3 句：背景是什么、决定了什么、为什么}

就这些。ADR 可以只是一段话。价值在于记录"做了决定"和"为什么"，不在于填满章节。
```

## 可选章节

仅当真正增加价值时才加，多数 ADR 不需要：

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）—— 决策被重新审视时有用
- **Considered Options** —— 被拒绝的备选方案值得记住时
- **Consequences** —— 非显而易见的下游影响需要标注时

带可选章节的完整模板：

```markdown
---
status: accepted
---

# {决策的简短标题}

{1-3 句：背景、决定、为什么}

## Considered Options（可选）

- 选项 A：{描述} —— {拒绝理由}
- 选项 B：{描述} —— {拒绝理由}

## Consequences（可选）

- {非显而易见的下游影响}
```

## 什么算数（值得写 ADR 的决策）

- **架构形态**："用 monorepo"、"写模型 event-sourced、读模型投影到 Postgres"
- **上下文间集成模式**："Ordering 与 Billing 通过领域事件通信，非同步 HTTP"
- **带锁定的技术选择**：数据库、消息总线、认证提供商、部署目标（不是每个库，只是换掉要一个季度的）
- **边界与范围决策**："客户数据归 Customer 上下文所有，其他上下文仅按 ID 引用"
- **故意偏离显见路径**："用手写 SQL 而非 ORM，因为 X"
- **代码中不可见的约束**："因合规不能用 AWS"、"响应时间必须 < 200ms"
- **非显而易见的拒绝**："考虑过 GraphQL 选了 REST，理由是..."
