# spec-grilling —— 核心访谈循环

对计划、决策或想法进行 relentless 访谈，直到每个决策树的分支都被解决。这是 spec-clarify 编排器在阶段 ① 调用的核心纪律。

## 访谈铁律（不可妥协）

### 1. 一次一问

每次只抛一个问题，等用户回答后再问下一个。问题堆在一起用户记不住，对齐效率反而下降。若你发现自己想一次抛多个问题，停下，拆成单问序列。

### 2. 给推荐答案

每个问题都给出你推荐的答案及理由。用户可以采纳、修改或否决——但不要抛空问题让用户从零思考。推荐答案要简短，避免长篇大论淹没问题本身。

### 3. 事实 vs 决策

- **事实（Facts）**：能通过探索环境（文件系统、代码库、工具、文档）查到的，自己查，不要问用户。例：现有代码模式、依赖版本、当前实现、目录结构。
- **决策（Decisions）**：需要用户拍板的，逐个问、等答案。例：架构选择、功能范围、技术栈、边界划定、优先级取舍。

判断标准：如果"查一下就知道"，就去查；如果"只有用户能决定"，才问。

### 4. 走遍决策树

沿决策树每个分支深入，逐个解决依赖关系，不要遗漏边界情况和边缘场景。一个决策往往依赖另一个决策——先解决被依赖的，再解决依赖方。

### 5. 共识前不行动

未得到用户明确确认"已达成共识"前，不写任何**业务**实现代码、不创建**业务**文件。访谈结束时暂停，汇总后明确问用户，等确认。

"业务"二字是关键限定——throwaway prototype（用完即弃原型）不是业务代码，受原型隔离三原则约束，可在访谈中有条件使用（见下节"throwaway prototype"）。

## throwaway prototype（用完即弃原型）

访谈中途为厘清某个具体设计问题（状态机转移、业务规则边界、UI 选型），可构建 throwaway prototype 验证假设。它不是业务代码，是访谈工具——性质等同于 agent 在脑内推演，只是把推演外化为可运行形式。

### 原型隔离三原则（同时满足才合法）

1. **目的明确**——仅为厘清当前访谈中的某个具体设计问题，非实现功能。"先写个完整功能看看效果"不算原型，是实现。
2. **位置隔离**——prototype 放在 `docs/prototypes/`，**绝不放 `src/`**（业务代码目录）。位置让"这是原型不是业务代码"在文件系统层面可见。
3. **生命周期标注**——prototype 文件顶部标注 `<!-- THROWAWAY PROTOTYPE: 验证完毕后删除，不纳入业务代码 -->`。进入阶段 ② 前，必须删除所有 prototype 文件。

任一缺失即违反不变式 I6。

### 允许与不允许的场景

- **允许**：状态机转移复杂，口头推演不清 → terminal prototype 验证；业务规则有边界冲突 → prototype 跑边缘 case；UI 有多个候选方案 → 多 UI 变体 prototype 对比。
- **不允许**："先写个完整功能看看效果"（这是实现）；"prototype 先放着以后用"（违反生命周期标注，必须删）。

### 与用户控制权

是否需要 prototype 由用户决定——你在访谈中提议，用户同意才建。prototype 不是绕过门禁的通道，是提升共识质量的工具。

## 联动 spec-modeling（边谈边写）

访谈不是纯对话——敲定的内容立即落地为文档。你不亲自写文档，所有写入交给 `spec-modeling`，你只负责触发：

- **术语敲定** → 触发 `spec-modeling` 写入 `CONTEXT.md`，不批量。
- **用户提出一条要求** → 主动提议分类（定为不变式/需求/边界/ADR），给推荐分类+理由，由用户拍板。判断不变式时用精炼三问（恒成立/违反即灾难/可证伪），详见 `SPEC-FORMAT.md` 的"写入前自检"子节。
- **需求/边界/不变式明确** → 触发 `spec-modeling` 按已确定的分类写入 `SPEC.md` 对应段落（`spec-modeling` 不重新做分类决策）。
- **出现符合 ADR 三条件的决策**（难撤销 + 令人惊讶 + 真实权衡）→ 提示用户并触发 `spec-modeling` 写 `docs/adr/`。

边谈边写的好处：访谈结束文档也就绪，阶段 ② 的出口门禁自然满足。

## 问题质量标准

- 每个问题指向**具体决策点**，不泛泛而谈。
- 优先问**边界**（"这个功能不做 X 吗？"）和**不变式**（"系统在任何情况下都要保证 Y 吗？"），而非只问正面需求。显式的 no 和显式的 yes 同样有价值。
- 用**具体场景**压力测试：构造边缘场景，迫使用户精确划定概念边界。例："当订单总额为负时，系统应当怎么处理？"

## 访谈节奏控制

- 发现自己想一次抛多个问题时，停下，拆成单问序列。
- 用户回答模糊时，先锐化术语（联动 `spec-modeling` 挑战 glossary），再继续。
- 避免在同一个已解决的决策上反复绕——解决了就推进下一个分支。
- **收敛判定**：当连续 2 个分支未引出新决策点（回答未产生新的"取决于"或"但是"），主动汇总已有决策清单并询问用户是否达成共识。这防止决策树无限细分消耗用户注意力。

## 访谈结束

1. **汇总**：清单形式列出本次解决的所有决策点。
2. **确认共识**：明确问用户："我们是否已达成共识，可以进入实现阶段？"
3. **交接**：得到肯定答复后，控制权交回 `spec-clarify` 推进到阶段 ② 出口检查。
